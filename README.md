# Movie Scene Battle Analyzer

`movie_scene_battle_analyzer` crawls [Movie Scene Battles](https://moviescenebattles.blogspot.com), normalizes Blogspot posts, and produces snapshot JSON artifacts used by analysis and a static stats page.

## Intent and architecture

The project keeps two deployable artifacts in sync:

- `data/moviescenebattles_dataset.json`: full crawl payload (`posts` + computed `stats`)
- `data/site_stats.json`: smaller page-facing payload used by `index.html`

The key design constraint is consistency: `site_stats.json` must always match the dataset-derived projection checked by `scripts/verify_site_snapshot.py`.

## Public interfaces

### CLI (`python3 -m movie_scene_battle_analyzer`)

```bash
python3 -m movie_scene_battle_analyzer \
  --max-posts 500 \
  --output data/moviescenebattles_dataset.json
```

Options:

- `--max-posts` (default: `500`)
- `--include-content` (stores `content_text`)
- `--output` (default: `data/moviescenebattles_dataset.json`)

### Python API

```python
from movie_scene_battle_analyzer import crawl_moviescenebattles, save_dataset

dataset = crawl_moviescenebattles(
    max_posts=300,
    include_content=False,
    page_size=150,
    timeout=30,
)
save_dataset(dataset, "data/moviescenebattles_dataset.json")
```

`crawl_moviescenebattles(...)` constraints:

- `max_posts > 0` (raises `ValueError` otherwise)
- `page_size > 0` (raises `ValueError` otherwise)
- Uses Blogspot feed pagination via `start-index` and `max-results`

## Data model and JSON contracts

### `BattlePost`

Normalized post fields include: `post_id`, `title`, `url`, timestamps, `comment_count`, `categories`, `word_count`, and optional `content_text`.

Comments are read from Blogspot `thr$total`; word counts are computed from HTML-stripped post text.

### `SiteStats`

Aggregate fields:

- `total_posts`, `total_comments`
- `average_comments_per_post`, `average_words_per_post`
- `posts_with_explicit_matchup`
- `posts_by_year`, `top_categories`, `most_commented_posts`
- `last_post_update`, `crawl_completed_at`

`posts_with_explicit_matchup` counts titles matching `vs`, `vs.`, `versus`, or `v` (case-insensitive).

### `CrawlDataset` (`data/moviescenebattles_dataset.json`)

Top-level shape:

- `site_title`
- `site_url`
- `posts` (`BattlePost[]`)
- `stats` (`SiteStats`)

### Site stats payload (`data/site_stats.json`)

Generated by `scripts/build_site_snapshot.py` with this required shape:

```json
{
  "site_title": "...",
  "site_url": "...",
  "generated_from_posts": 0,
  "stats": { "...": "SiteStats fields" }
}
```

`scripts/verify_site_snapshot.py` validates both file shapes and enforces:

- `generated_from_posts == len(dataset.posts)`
- `site_stats.stats == dataset.stats`
- top-level site metadata matches dataset metadata

## Operational runbook

### Build deployable snapshot artifacts

```bash
python3 scripts/build_site_snapshot.py
```

This script currently crawls with `max_posts=1000` and writes:

- `data/moviescenebattles_dataset.json`
- `data/site_stats.json`

### Verify artifacts before/after changes

```bash
python3 scripts/verify_site_snapshot.py
```

If verification fails with an out-of-sync error, regenerate with the build command above.

### Static page expectations

`index.html` fetches `data/site_stats.json` (with `cache: "no-store"`) and renders:

- headline metadata (`site_title`, `generated_from_posts`)
- top-level `stats` values/cards
- `top_categories` and `most_commented_posts` lists

## CI workflows

### Verify Site Snapshot (`.github/workflows/verify-site-snapshot.yml`)

- Triggers on PRs targeting `main` and manual dispatch
- Uses Python `3.12`
- Runs `python3 scripts/verify_site_snapshot.py`

### Refresh Site Snapshot (`.github/workflows/refresh-site-snapshot.yml`)

- Triggers daily at cron `20 6 * * *` and on manual dispatch
- Rebuilds and verifies snapshot artifacts
- Uses `peter-evans/create-pull-request@v7` to update branch `ci/refresh-site-snapshot`

## Setup and troubleshooting

- Runtime dependencies: Python standard library only (no third-party packages required)
- If crawl output has fewer posts than `--max-posts`, the upstream feed likely has no additional entries
- For slow/unreliable network conditions, use the Python API to tune `page_size` / `timeout`
